## 1. 升级核心逻辑

### 1.1 一句话定位
从「拍题→批改→存错题」升级为「批改→打标签→算掌握度→生成计划→测验验收→反馈更新」的**自学习闭环系统**，每次做题的数据都在喂养更精准的下一步学习计划。

### 1.2 v1 → v2 对比
| 维度 | v1.0 现状 | v2.0 目标 |
|------|-----------|-----------|
| 输出结构 | 纯文本批改 | 结构化 JSON（对错+知识点+错型） |
| 数据沉淀 | 错题原文+错因 | 错题+知识点+错型+掌握度+计划 |
| 用户看到 | 错题列表 | 薄弱点热力图+学习计划+测验 |
| 闭环 | 一次性的存档 | 数据循环驱动下一轮计划 |
| 价值边界 | 批改工具 | 自适应学习助理 |

## 2. 关键认知（来自分析）

**改动成本远比想象低。** 最关键一步是改造批改时的 prompt，让 Claude 在返回对错同时，多返回「知识点名称」和「错误类型」两个结构化字段。这一步几乎不增加代码量，却是所有新功能的数据基础——热力图、计划、测验都建在其上。

**因此实施顺序锁定为：先做标签化（②）解锁数据，再逐层向上建功能。**

## 3. 数据层设计

### 3.1 现有表（v1）
`questions(id, question, error_analysis, subject, image_url, created_at)`

### 3.2 v2.0 新增/改造

**改造 `questions` 表**（向后兼容，新增可空字段）：
```
+ is_correct        bool        是否全对
+ knowledge_point   text        知识点名称（如"两位数进位加法"）
+ error_type        text        错误类型（计算失误/概念不清/审题错误/方法错误）
+ mastery_delta     int         本次对掌握度的影响分
+ user_id           uuid        归属用户（配合登录）
```

**新增 `knowledge_points` 表**（知识点字典 + 掌握度统计）：
```
id, name, subject, parent_id(父知识点),
mastery_level int(0-100),   掌握度
total_count int,            该知识点累计做题数
correct_count int,          正确数
last_practiced_at timestamptz,
user_id
```

**新增 `study_plans` 表**（学习计划）：
```
id, user_id, title, target_knowledge_point,
current_mastery, target_mastery,
status(pending/studying/done),
steps jsonb,                AI 拆解的步骤
created_at, due_date
```

**新增 `test_records` 表**（测验记录）：
```
id, user_id, plan_id, knowledge_point,
score, total, passed bool, created_at
```

完整建表 SQL 见 `docs/DEPLOYMENT.md` v2.0 章节。

## 5. 批改 prompt 设计（v2.0 核心）

将提示词从「四段式文本」改为「要求 JSON 输出」，前端拿到结构化字段后既能展示又能入库。

期望 Claude 返回结构：
```json
{
  "is_correct": false,
  "error_type": "计算失误",
  "knowledge_point": "两位数进位加法",
  "error_spot": "第二步 27+38 未进位",
  "correct_solution": "27+38=65...",
  "analysis": "对进位规则掌握不牢...",
  "knowledge_tags": ["进位加法", "竖式计算"]
}
```
带容错：Claude 偶尔不严格 JSON，前端用正则剥离 ```json 代码块再 parse。

## 6. 掌握度算法（草案）

某知识点 `mastery_level`：
- 每做一题：正确 +α、错误 −β，α/β 随历史题量衰减（避免早期一题定论）
- 初始 50，区间 [0, 100]
- mastery_level < 40 列为「薄弱」，触发计划生成
