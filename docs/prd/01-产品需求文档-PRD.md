## 1. 背景与目标

### 1.1 背景
公司生产环境已部署 **观测云（Guance）** 作为统一可观测性平台，日常产生大量告警（指标异常、日志错误、链路异常等）。当前告警处理依赖运维/SRE人员人工排查，存在以下痛点：

- 告警发生后，根因分析（RCA）完全依赖人工经验，效率低、结论主观、缺乏沉淀；
- 告警截图、日志散落在各处，缺乏统一的分析入口；
- 历史故障分析结果无法复用，重复问题需要重新排查；
- 缺少结构化、可追溯的RCA报告输出，不利于故障复盘和考核。

### 1.2 目标
建设一套**告警RCA报告生成系统**，实现：

1. 支持运维人员手动提交"告警截图 + 告警日志"，进行单次独立分析；
2. 对接第三方AI大模型（如通义千问/DeepSeek/GPT系列等），自动生成结构化RCA报告；
3. 沉淀历史RCA报告，支持查询、检索与复用。

### 1.3 非目标（本期不做）
- 不做告警的自动实时监听与全自动触发分析（本期为"人工提交触发"模式）；
- 不做多告警关联的批量根因分析（仅支持单次/单告警分析）；
- 不替代观测云原生的监控告警能力，仅作为其下游分析工具。

## 2. 名词术语

| 术语 | 说明 |
|---|---|
| 观测云 / Guance | 现有生产环境可观测性监控平台 |
| RCA | Root Cause Analysis，根因分析 |
| 告警材料 | 用户上传的告警截图、告警日志文本等原始输入 |
| 分析任务 | 一次RCA报告生成请求，包含输入材料、调用记录、输出结果 |
| 大模型服务商 | 第三方AI大模型API提供方 |

## 3. 用户角色与使用场景

### 3.1 角色

| 角色 | 说明 | 权限 |
|---|---|---|
| 运维/SRE工程师 | 主要使用者，日常处理告警 | 提交分析、查看自己及团队历史记录 |
| 运维主管/技术负责人 | 复盘、审核 | 查看全部历史记录、导出报告、（可选）配置模型 |
| 系统管理员 | 系统配置 | 配置AI模型接入参数、管理用户权限、查看调用统计 |

### 3.2 典型场景

**场景一：单次故障排查**
运维工程师在观测云上发现一条CPU使用率突增告警，截图告警详情页面，并复制相关日志片段，进入RCA系统提交，几分钟内获得AI生成的可能根因、影响范围、处置建议。

**场景二：故障复盘**
故障处理完毕后，主管在系统中检索该服务近3个月的历史RCA报告，用于复盘会材料整理。

**场景三：知识复用**
新人遇到相似告警，可以在历史记录中搜索关键字（如服务名、错误码），参考历史分析结论。

## 4. 功能需求

### 4.1 功能架构总览

| 模块 | 子功能 |
|---|---|
| 材料提交模块 | 截图上传、日志粘贴/上传、基础信息录入 |
| AI分析引擎 | Prompt编排、模型调用、结果解析 |
| 历史记录模块 | 报告列表/检索、报告详情、导出（PDF/Word） |
| 系统管理 | 模型配置/用户权限/日志审计 |

### 4.2 模块一：告警材料提交（单次分析入口）

| 编号 | 功能点 | 需求描述 | 优先级 |
|---|---|---|---|
| F1.1 | 截图上传 | 支持拖拽/点击上传告警截图，格式支持 png/jpg/jpeg，单文件≤10MB，单次最多上传5张 | P0 |
| F1.2 | 日志输入 | 支持两种方式：①文本框直接粘贴日志；②上传日志文件（.log/.txt/.csv），单文件≤5MB | P0 |
| F1.3 | 基础信息录入 | 手动补充：告警名称、所属服务/应用、告警级别、发生时间、当前状态等结构化字段 | P0 |
| F1.4 | 图片OCR/理解 | 截图通过具备视觉理解能力的多模态模型直接解析 | P1 |
| F1.5 | 提交前预览 | 提交前可预览已录入的截图、日志、信息，支持删除重传 | P1 |
| F1.6 | 分析进度提示 | 提交后展示分析进行中状态（排队中/分析中/已完成/失败） | P0 |
| F1.7 | 失败重试 | AI调用失败时，支持一键重新发起，不需要重新上传材料 | P1 |

### 4.3 模块二：第三方AI大模型对接

| 编号 | 功能点 | 需求描述 | 优先级 |
|---|---|---|---|
| F2.1 | 模型接入配置 | 系统管理员可配置一个或多个大模型服务商的API Key、Endpoint、模型版本 | P0 |
| F2.2 | 多模型支持 | 架构上预留多模型适配层，支持接入如通义千问、DeepSeek、Kimi、GPT系列、Claude等 | P0 |
| F2.3 | Prompt模板管理 | 系统内置RCA分析的Prompt模板，管理员可维护和调整模板 | P0 |
| F2.4 | 结构化输出解析 | 要求模型以固定JSON/结构化格式返回，系统解析后渲染为可读报告 | P0 |
| F2.5 | 调用超时与降级 | 设置合理超时时间（如60-90秒），超时/失败给出明确提示 | P0 |
| F2.6 | Token/调用成本统计 | 记录每次调用的Token消耗，管理员可查看用量统计 | P1 |
| F2.7 | 数据安全 | 上传至第三方大模型的日志/截图需支持敏感信息脱敏 | P0 |

### 4.4 模块三：RCA报告展示与管理

| 编号 | 功能点 | 需求描述 | 优先级 |
|---|---|---|---|
| F3.1 | 报告标准结构 | 报告包含：告警基本信息、现象摘要、AI分析的可能根因、影响范围评估、处置建议、原始材料回溯 | P0 |
| F3.2 | 报告详情页 | 结构化展示，支持根因逐条展开查看依据 | P0 |
| F3.3 | 人工标注 | 使用者可对AI结论进行"采纳/部分采纳/不准确"标注并补充实际根因 | P1 |
| F3.4 | 报告导出 | 支持导出为PDF、Word格式 | P1 |

### 4.5 模块四：历史记录

| 编号 | 功能点 | 需求描述 | 优先级 |
|---|---|---|---|
| F4.1 | 历史列表 | 以列表形式展示所有历史分析任务 | P0 |
| F4.2 | 检索与筛选 | 支持按时间范围、服务名、告警级别、关键字、提交人筛选 | P0 |
| F4.3 | 详情回溯 | 点击进入可查看完整报告及当时提交的原始材料 | P0 |
| F4.4 | 权限隔离 | 普通用户默认查看本人提交记录，主管/管理员可查看全量 | P1 |
| F4.5 | 统计看板（可选） | 按服务/时间维度统计告警RCA数量、根因分类分布 | P2（二期） |

## 5. 交互流程（用户视角）

```
登录系统 → 新建分析任务 → 上传告警截图/日志 → 补充告警基础信息 → 提交
    → AI分析中（排队中/分析中/已完成/失败）
    → 成功：生成结构化RCA报告 → 查看/导出报告 → 标注结论准确性 → 自动归档至历史记录
    → 失败：提示失败原因，支持重试
```

## 6. 非功能需求

### 6.1 性能要求
- 单次AI分析任务响应时间目标：≤90秒；
- 支持并发提交，任务排队处理。

### 6.2 安全与权限
- 系统需接入公司统一身份认证（SSO）；
- 上传材料、生成报告需按角色做数据隔离；
- 操作日志留存，满足审计要求。

### 6.3 可用性
- 系统本身需具备基础的高可用部署能力；
- AI服务商单点故障时，需有明确报错提示。

### 6.4 数据安全
- 上传的截图、日志在传输与存储环节均需加密；
- 明确数据保留周期；
- 涉及第三方大模型调用的数据出域问题需安全评审。

---

## 7. 验收标准

1. 用户可成功提交截图+日志并获得结构化RCA报告，端到端全流程无阻断性Bug；
2. 报告字段完整，包含现象、根因、影响范围、处置建议四大核心板块；
3. 历史记录可正确检索、回溯到原始提交材料；
4. AI模型调用异常场景均有清晰的用户侧提示，无静默失败；
5. 涉及敏感信息的处理逻辑需在评审阶段与安全团队确认并落地；
6. 通过基本的权限隔离测试。

## 8. 待确认问题清单

| 序号 | 问题 | 影响 |
|---|---|---|
| 1 | 本期是"人工提交触发分析"还是"观测云告警自动推送触发"？ | 决定是否需要对接观测云Webhook/OpenAPI |
| 2 | 第三方大模型具体选型？是否要求私有化部署/不留存数据？ | 直接影响安全合规方案与架构设计 |
| 3 | 告警日志/截图中的敏感信息是否需要脱敏后再发送给大模型？ | 决定是否需要额外的脱敏中间层 |
| 4 | 历史记录的权限隔离粒度（个人/团队/全公司）？ | 影响权限模型设计 |
| 5 | 是否需要与观测云做账号/告警数据的直接集成？ | 影响是否需要观测云侧API/插件开发配合 |
| 6 | 报告导出格式和模板是否有既定企业规范？ | 影响报告导出模块设计 |
| 7 | 预计日均/峰值分析请求量？ | 影响并发架构与成本预估 |
