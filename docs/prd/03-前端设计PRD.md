## 1. 设计目标与原则

### 1.1 设计目标
- 让运维/SRE人员在**最短路径内**完成"材料提交 → 获得报告"的核心操作
- 报告展示需符合**行业标准RCA报告结构**，便于直接用于复盘会、审计留档
- 历史记录检索效率优先，支持快速定位到某次故障

### 1.2 设计原则
1. **单任务聚焦**：提交页面一次只做一件事（新建一次分析）
2. **状态可感知**：AI分析是异步耗时操作（30-90秒），全程需要明确的进度反馈
3. **原文可回溯**：报告中的每一个结论旁边，都能追溯到支撑它的原始材料
4. **报告即交付物**：报告页面要达到"可以直接截图/导发给领导"的成熟度

## 2. 信息架构（IA）

```
观测云RCA报告系统
├── 首页 / 工作台
│   ├── 快速新建分析入口
│   └── 最近报告 / 待办提醒
├── 新建分析（提交流程）
│   ├── Step1 上传材料（截图+日志）
│   ├── Step2 补充告警信息
│   └── Step3 提交确认 → 分析中 → 报告结果
├── 历史记录
│   ├── 列表 / 筛选 / 检索
│   └── 报告详情页
├── 报告详情页（独立可分享路由）
├── 系统管理（仅管理员可见）
│   ├── AI模型配置
│   ├── Prompt模板管理
│   ├── 用户与权限
│   └── 调用统计 / 审计日志
└── 个人中心（账号信息、通知设置）
```

> 导航建议采用左侧固定导航栏（工作台 / 新建分析 / 历史记录 / 系统管理）

## 3. 标准RCA报告结构

| 板块 | 说明 | 是否覆盖 |
|---|---|---|
| 1. 事件概述 | 告警名称、级别、影响服务、时间、状态 | ✅ |
| 2. 时间线 | 从告警触发到处置完成的关键时间节点 | ✅（部分手动补充） |
| 3. 影响范围评估 | 受影响的服务/用户/业务范围 | ✅ |
| 4. 根因分析 | 直接原因、根本原因、诱发因素，按置信度排序 | ✅（AI生成核心输出） |
| 5. 分析依据 | 支撑根因结论的原始材料引用 | ✅ |
| 6. 处置建议 | 短期止损+长期改进措施，标注优先级 | ✅ |
| 7. 经验总结 | 可复用的知识点 | ⭕ 二期 |
| 8. 关键指标 | MTTD/MTTR等 | ⭕ 二期 |

## 4. 页面详细设计

### 4.1 工作台（首页）
- 顶部：醒目的"新建分析"CTA按钮
- 中部：最近5条报告卡片（告警名称、级别色块、生成时间、根因摘要）
- 右侧：本周分析数量、根因分类分布简易统计图
- 空状态：展示引导插画+"开始你的第一次RCA分析"

### 4.2 新建分析 — Step1：上传材料

| 区域 | 设计要点 |
|---|---|
| 截图上传区 | 拖拽区域，支持点击选择；缩略图网格展示，可预览/删除/重排序；上限5张、单张≤10MB |
| 日志输入区 | Tab切换"粘贴文本/上传文件"；支持≥5000字符，显示字符/行数统计 |
| 校验提示 | 截图和日志至少提供一项才可进入下一步 |

### 4.3 新建分析 — Step2：补充告警信息

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 告警名称 | 文本 | 必填 | |
| 所属服务/应用 | 下拉+可搜索 | 必填 | 避免手输不一致 |
| 告警级别 | 单选（紧急/高/中/低） | 必填 | 对应报告影响分级 |
| 告警发生时间 | 日期时间选择器 | 必填 | 用于时间线 |
| 首次发现/响应时间 | 日期时间选择器 | 选填 | 用于计算MTTD |
| 当前状态 | 单选（未恢复/已恢复/处理中） | 必填 | |
| 恢复时间 | 日期时间选择器 | 条件必填 | 用于计算MTTR |
| 补充说明 | 多行文本 | 选填 | |

### 4.4 新建分析 — Step3：提交确认与分析中状态

**分析中状态设计（重点）**

分阶段进度条：
```
① 材料解析中 → ② AI根因分析中 → ③ 报告生成中 → 完成
```
- 每个阶段配合简短文案，降低焦虑感
- 超过30秒提示"可关闭页面，完成后通知"
- 失败状态明确展示原因分类，提供"重新分析"按钮

### 4.5 报告详情页（核心页面）

采用**卡片分区+锚点导航**布局：

| 板块 | 内容 |
|---|---|
| 顶部 | 告警名称、级别色块、状态、生成时间；操作栏：导出PDF/Word/分享 |
| 1. 事件概述 | 服务/应用、发生/发现/恢复时间、当前状态 |
| 2. 时间线 | 横向时间轴组件，节点打点+间隔时长 |
| 3. 影响范围评估 | 严重程度标签+影响描述 |
| 4. 根因分析 | 按置信度列出候选项，含查看依据按钮 |
| 5. 分析依据 | 截图缩略图+引用日志片段高亮 |
| 6. 处置建议 | 分"短期止损"和"长期改进"，Checklist样式 |
| 7. 人工反馈区 | 采纳/部分采纳/不准确 + 补充说明 |

**设计要点**
- 根因分析是视觉重心，主题色高亮，置信度可视化
- "依据联动"：点击根因旁【查看依据】，自动跳转到对应截图/日志

### 4.6 历史记录列表页

| 设计要点 | 说明 |
|---|---|
| 筛选栏 | 时间范围、服务、级别、状态、提交人，常驻顶部多条件组合 |
| 搜索框 | 按告警名称/根因关键字模糊搜索 |
| 列表形式 | 表格：告警名称、服务、级别、状态、提交时间、提交人、操作 |
| 级别可视化 | 色块/标签区分优先级 |

### 4.7 系统管理页面（管理员）
- **AI模型配置**：模型列表+新增/编辑/测试连通性
- **Prompt模板管理**：文本编辑器+预览+版本历史
- **用户与权限**：角色分配
- **调用统计**：Token消耗、调用次数趋势、失败率图表

## 5. 关键交互状态清单

| 场景 | 需设计的状态 |
|---|---|
| 材料上传 | 上传中/成功/失败（可重试）/超出限制提示 |
| AI分析 | 排队中/分析中（分阶段）/成功/超时失败/服务异常失败 |
| 历史列表 | 加载中骨架屏/正常数据/筛选无结果/首次无历史记录 |
| 权限相关 | 无权限访问提示页 |
| 网络异常 | 全局网络错误提示条，不打断已填写表单 |

---

## 6. 视觉与组件规范

- **色彩语义**：告警级别沿用观测云颜色（紧急=红、高=橙、中=黄、低=灰/蓝）
- **组件复用**：时间轴组件、置信度可视化、Checklist组件作为基础组件沉淀
- **响应式**：核心场景PC端，移动端保证基本可读

## 7. 与后端PRD的联动确认项

| 序号 | 前端设计依赖点 | 需要确认 |
|---|---|---|
| 1 | 时间线字段（发生/发现/恢复时间）是否必填？ | 是否用于计算MTTD/MTTR |
| 2 | 根因"依据联动"（跳转到截图/日志高亮位置） | AI输出需包含引用定位信息 |
| 3 | 分析中的分阶段进度展示 | 后端是否能提供真实阶段性状态 |
| 4 | 人工反馈区 | 反馈数据是否需要展示给管理员统计 |
| 5 | 调用统计图表所需数据字段 | 需要对应统计接口字段定义 |
